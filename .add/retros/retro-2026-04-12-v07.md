## Agent Self-Assessment

**ADD Methodology Effectiveness: 5.8 / 9.0**

Strong: community PR handling, version-bump automation, security hardening infrastructure. Weak: v0.7.0's scale warranted a spec and didn't get one. `/add:cycle` continues to be bypassed even in ADD's own dog-food. The compliance rule is now the strongest mechanism to force discipline — but it just shipped, so validation is next arc.

**Swarm Effectiveness: 8.1 / 9.0**

5-agent competing-swarm pattern for the install/Codex review worked exceptionally well. Each agent stayed in its lane (explicit "don't discuss X" briefing), produced findings the others missed, and synthesis was straightforward because conflicts were minimal and complementary. Validates the pattern promoted to Tier 1 knowledge in v0.6.0.

## What Went Well

- **Bypass-approvals mode with explicit task list** — scaled to a 9-step, 3-release sprint without ambiguity
- **Community PR merge policy** (merge as-submitted, refactor later with co-authorship) — got 3 contributors shipped and credited same day
- **Extraction without breaking consumers** — `plugins/add/` stayed marketplace-consumable throughout; Claude install path unchanged
- **Compile drift-check as a design forcing function** — committing both source and generated output stays honest because CI enforces the relationship
- **Competing-swarm review on the plugin itself** — 5 angles × ~90 seconds per agent produced a denser review than any single-pass would have

## What Didn't Work

- **No `specs/` artifact for the v0.7.0 architecture work.** Spec-before-code violated. The scale (runtime abstraction, CI gates, security hardening) warranted a spec that could be referenced in commits. Future multi-release arcs should spec up front.
- **`/add:cycle` bypassed again.** Third consecutive major ADD arc (v0.5, v0.6, v0.7) with no cycle artifact. The cycle command is too heavy for sprint velocity OR the arcs are too fast for the command's abstraction. Either way, a gap.
- **Schema validation caught YAML edge cases that compile.py would have silently propagated.** Good the schema exists; bad that we didn't notice the issue until schema-check surfaced it.
- **Retro was reactive, not scheduled.** The rule that just shipped is supposed to force retros; it couldn't force this one because it just shipped. Validates the need for the rule; doesn't validate the rule's effectiveness yet.

## Process Mutations (for v0.7.1+)

1. **Spec-first for any arc with ≥3 release-worthy changes.** If the plan has multiple independently-shippable pieces, a `specs/v{X}.{Y}.0-arc.md` is warranted regardless of maturity.
2. **`/add:cycle` becomes `/add:arc`** (rename candidate for v0.8.0)? The word "cycle" implied time-boxed planning that hasn't fit the actual workflow. "Arc" maps to how the work actually flowed in v0.5-v0.7.
3. **Auto-write a release-checkpoint learning on every `gh release create`.** No manual step needed; the release itself is the trigger. Currently the release shipped but the structured learning didn't get written — that's a hole.
