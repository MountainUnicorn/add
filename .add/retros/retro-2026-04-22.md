## Agent Observations

### What worked well
- **Parallel swarm dispatch (12× across the session)** scaled cleanly. Each swarm was given a tight scope, distinct lane, and explicit out-of-scope list. No swarm produced output that overlapped with another's lane. Total agent-side context spend across 12 swarms ≈ 530K tokens; my main thread stayed under 200K throughout.
- **Cycle/Tranche framing** for multi-step work (Tranche 1/2/3 for website cleanup) created natural commit boundaries and let the user veto/approve at each step. Made it possible to do 75 file changes across multiple commits without losing thread.
- **CI gates as guardrails** — frontmatter validate + compile-drift + fixture tests caught zero regressions across the arc. Running them before every commit caught two near-misses (after the website extraction, after the v0.8 bump).
- **`gh api` directly for Pages cutover** — faster and more atomic than UI-driven domain reassignment. The `gh api -X PUT .../pages -f cname=` pattern is reusable.
- **`git filter-repo` for the website extraction** — clean preservation of 38 commits with zero history loss. Worth the one-time `brew install` cost.
- **Proactive verification on risky external actions** — polling `curl` until expected content appeared (Pages deploys, domain cutover, blog post live) caught the build-cache window without false success claims.

### What was harder
- **`reports/dashboard.html` drift** — the v0.7.3 snapshot I generated mid-session was stale by end of session (after v0.8.0 ship). Surgical update worked but reinforced that snapshots want explicit regeneration discipline. The `/add:dashboard` skill should probably refuse to overwrite if the on-disk version is newer than the data source it would reflect, OR auto-regenerate after every milestone change.
- **`.add/config.json` carried `current_milestone: "M2-install-and-safety"` for an unknown duration** — the milestone file never existed on disk, no validator caught the drift. Would have benefitted from a `/add:verify` gate that checks every config-referenced file actually exists.
- **GitHub Pages on the old repo couldn't be deactivated via API** — "Deactivating GitHub pages for this repository is not allowed." Required leaving the stale `mountainunicorn.github.io/add/` URL as cached content. Worth flagging for any future "deprecate-and-redirect" repo work.
- **The agent-architecture.svg dependency** in README.md was almost forgotten during the website extraction — caught it before the final commit, but one more cross-reference and it would have been a broken-image post-deploy. Suggests an `--audit-references` flag on extraction-style operations.

### What surprised me
- **Anthropic's 2026 Trends Report explicitly endorsing spec-first development as the highest-impact practice** — the methodology ADD has been making since v0.1 is now mainstream-validated by the platform vendor. This is a tailwind to lean into in v0.9 messaging.
- **Codex CLI shipped Skills, sub-agents, hooks, and a plugin marketplace** in the last 3-6 months — ADD's Codex adapter is now compiling to a deprecated target. The "multi-runtime promise" requires real catch-up work in v0.9, not just maintenance.
- **GitHub Pages on private repos works without a Pro/Team plan** as of 2026-04 — no plan upgrade was needed for the website extraction. Saved a potentially blocking step.
- **`git filter-repo` worked first-try** with `--subdirectory-filter` — cleaner than expected, no manual history rewriting.

## Decisions for the Methodology Itself

These are observations about ADD-the-methodology that emerged from dog-fooding it this session.

1. **The v0.7 `core/` + `compile.py` split is the durable architecture.** Not just a refactor — a methodology decision that pays compounding dividends. The version bump touched `core/VERSION` + 5 hand-edited UI surfaces + 1 compile run; the pre-v0.7 version was a 7-file mechanical checklist. The website extraction was clean precisely because `core/` didn't depend on `website/`. Memory's `compile_flow.md` captures this for future sessions.
2. **The `/add:dashboard` snapshot pattern needs a "stale" signal.** Generated dashboards drift the moment underlying state changes. Either auto-regenerate after milestone/release changes, or display a "data-as-of" timestamp + diff-against-current indicator.
3. **Parallel agent swarms are now a first-class ADD pattern, not just an efficiency hack.** This session used 12. The methodology should have explicit guidance on when to use them (research with distinct lanes, parallel spec drafting on disjoint files) vs when not (anything requiring tight serial reasoning, anything where 1 swarm's output blocks another's input).
4. **CHANGELOG `[Unreleased]` should differentiate "Pending for next release" from "Deferred indefinitely"** — the prior 3-bullet "Pending for v0.9.0" stub was indistinguishable from "we'll do this someday." The M3 plan now uses a table for in-scope and an explicit Deferred list for out-of-scope. This pattern could ship as a CHANGELOG template update.
5. **Memory entries about *why* a thing was done outlive memory entries about *what* was done.** The ones I wrote this session that will hold up best (`website_repo.md`, `compile_flow.md`) lead with rationale and explicit "how to apply" guidance. The format from MEMORY.md's instructions is right; reinforcing the discipline.

## Promotions to Workstation Library

The following learnings are scoped beyond just ADD and should be promoted to `~/.claude/add/library.json` (Tier 2):

- **"You choose A or B" delegation pattern** — when user explicitly delegates a tactical choice, execute the chosen path; don't return another round of options.
- **"Audit and plan" two-phase pattern** — when user separates audit from execute, wait for explicit confirmation before any write operations even if the plan is small.
- **Parallel agent swarms with bracketed lanes** — for research and spec generation across 4+ distinct topics, dispatch in a single message with explicit out-of-scope lists per agent. Total agent-side context across 12 swarms easily fits within session budget.
- **GitHub Pages cutover via `gh api -X PUT .../pages`** — atomic, faster than UI flow, preserves TLS cert across repo changes.
- **`git filter-repo --subdirectory-filter` for repo splits** — preserves history cleanly; one-time `brew install` cost.
