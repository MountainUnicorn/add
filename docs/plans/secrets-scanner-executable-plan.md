**Spec:** `specs/secrets-scanner-executable.md`

**Status:** Draft
**Target Release:** v0.9.2
**Milestone:** none (standalone v0.9.x hardening)
**Closes:** F-014 from `specs/plugin-family-release-hardening.md`

## 1. Overview

Convert the secrets-handling story from "documented intent" to "enforced contract" by shipping a small executable scanner (`core/lib/scan-secrets.sh`) that any caller — deploy skill, verify gate, future hook, human at the shell — invokes to scan staged content. Findings emit on stdout in a stable format; non-zero exit blocks the caller. The script is the single point of truth for "does this commit contain a secret?" The catalog moves from markdown-only into a sibling JSON file (`core/security/secret-patterns.json`) so shell can parse it without `jq` in the hot path.

This plan covers the runtime-neutral source changes in `core/`, plus three skill/rule edits and a hooks.json entry. `scripts/compile.py` regenerates `plugins/add/` and `dist/codex/` automatically — generated output is never edited by hand.

The work is sequenced RED-then-GREEN: write fixtures first against the *intended* CLI surface, watch them fail, write the script to make them pass, then wire callers.

## 5. Phased Breakdown

### Phase 1 — RED: Fixtures and Test Runner (~0.4d)

Build the failing tests first against the spec's intended CLI surface. The script does not exist yet; every test should fail with `command not found` initially.

1. Scaffold `tests/secrets-scanner-executable/` with the synthesizer copied from `tests/secrets-handling/`. Keep the `<SYNTHESIZED:{NAME}>` placeholder pattern — the repo must never store a literal secret-shaped string (Swarm B v0.9.0 hit GitHub Advanced Security push-protection on this; the synthesizer dodges it).
2. Write `test-scan-secrets.sh` with the test cases from spec § 2.H AC-030. Each case: stage fixtures, invoke script, assert exit code + stdout shape.
3. Write `test-redaction-integrity.sh`: synthesize every positive fixture, run script, capture stdout+stderr, grep-assert no synthesized value appears.
4. Write `test-perf.sh`: generate 1,000 small text files, stage them, time the script, assert wall time < 2,000 ms.
5. Run the suite. All tests fail with `scan-secrets.sh: not found`. RED achieved.

### Phase 2 — GREEN: Catalog Migration and Script Body (~0.7d)

1. Generate `core/security/secret-patterns.json` from the existing markdown catalog. One pass: read § 1 of `core/knowledge/secret-patterns.md`, emit a JSON object per pattern with a fresh `SEC-NNN` code (alphabetic order: AWS=001, GITHUB=002, STRIPE=003, OPENAI=004, ANTHROPIC=005, JWT=006, PASSWORD_KV=007, PEM=008).
2. Write `scripts/validate-secret-patterns.py`: tiny drift checker. Reads both files, asserts name parity, compiles each regex.
3. Write `core/lib/scan-secrets.sh`. Implementation order:
   - Header: shebang, `set -euo pipefail`, usage block, exit-code documentation.
   - CLI parsing for `--paths`, `--all`, `--commit-msg-file`, `--allow`, `--audit-log`, `--max-bytes`, `--verbose`, `--help`.
   - Catalog reader: `awk`-based extractor that emits `code|name|regex` lines from the JSON (no `jq` dependency in the hot path; JSON is structurally simple so a small awk script handles it).
   - File-list resolution: default `git diff --cached --name-only --diff-filter=ACM`; `--paths` override; `--all` traversal.
   - `.secretsignore` loader and matcher (gitignore syntax — use `git check-ignore` if available, fall back to a literal-match implementation).
   - Per-file content scan loop: `grep -nE` per pattern, with safe-context entropy suppressions documented in `core/knowledge/secret-patterns.md` § 2.
   - Trailer-override parser: read `--commit-msg-file`, extract `[ADD-SECRET-OVERRIDE: ...]`, parse SEC-NNN codes.
   - Output formatter: redacted preview per AC-013, sorted by path/line.
   - Audit-log writer: JSONL append on `--audit-log`.
   - Exit logic: `0` if all findings overridden, `1` otherwise.
4. Iterate on the script until every Phase 1 fixture passes. GREEN achieved.

### Phase 3 — Wire Into Skills and Hooks (~0.3d)

1. Edit `core/skills/deploy/SKILL.md` § Pre-commit secrets gate: replace the prose-driven scan with a single block describing the script invocation. Keep the `--allow-secret` confirmation-phrase wrapper because that human-gated path is the parent spec's contract (`secrets-handling.md` AC-016).
2. Edit `core/skills/verify/SKILL.md`: insert Gate 4.6 between Gate 4.5 and Gate 5. Use the existing gate output template.
3. Edit `core/rules/secrets-handling.md` § Template + Deploy-Gate Contracts: rewrite the deploy-gate bullet to cite the script.
4. Edit `core/knowledge/secret-patterns.md`: add the executable-source pointer note above § 1.
5. Edit `runtimes/claude/hooks/hooks.json`: add a `PreToolUse` matcher for `Bash` matching `git push`, advisory only.
6. Update `scripts/compile.py` if needed to copy `core/security/` into both runtime outputs. The existing `core/lib/` copy rule should handle the new shell script automatically (verify by re-running `compile.py --check`).

## 6. TDD Strategy

### RED Phase
- Phase 1 above. Tests fail with command-not-found before the script exists.
- Fixtures use the existing synth-at-runtime pattern (placeholder strings expanded by the runner) so the repo never contains a literal secret-shaped string and GHAS push protection stays quiet.

### GREEN Phase
- Phase 2 above. Script body grows incrementally — start with a no-op that exits 0, add catalog reader, add per-file scan, add trailer parser, add redaction. Watch the fixture pass count climb.

### REFACTOR Phase
- Once green, factor out the catalog reader and the redaction formatter into shell functions. Add `--verbose` instrumentation. Audit for any code path that could echo a raw match value (this is what AC-015 + the redaction-integrity test exist to catch — never trust review alone).

### VERIFY Phase
- Run all fixture suites. Run perf harness on a representative laptop. Confirm exit codes match spec. Confirm redaction-integrity grep finds zero synthesized strings in any captured stream.

## 7. Risk Register

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| **GitHub Advanced Security push protection blocks fixture commits.** Swarm B hit this during v0.9.0 implementation; literal AWS-key-shaped strings in the repo trigger GHAS. | High | High (blocks PR merge) | Use the synth-at-runtime pattern proven in `tests/secrets-handling/`. The repo only contains `<SYNTHESIZED:AWS_ACCESS_KEY>` placeholders; the runner expands at exec time. Verified before the spec was written. |
| **False-positive rate on real-world repos.** Generic `password = "..."` and entropy heuristics may flag legitimate code. | Medium | Medium | Ship only `high`-confidence patterns as blocking by default. Entropy heuristic (`SEC-999`) is warn-only in v0.9.2 (Open Q-002). Document `--allow` and `.secretsignore` escape valves prominently. |
| **Perf budget violation on large diffs.** Per-file `grep -E` per pattern is O(files × patterns). 1,000 files × 8 patterns = 8,000 invocations. | Medium | Medium | Use one `grep -E -f <combined-patterns-file>` invocation per file (8x fewer process spawns). Time the perf test in CI; fail the build at > 2s. Add `--max-bytes` truncation for outliers. |
| **JSON parsing in pure shell is fragile.** Hand-rolling a JSON parser invites bugs. | Medium | Medium | The catalog is structurally trivial (flat array of flat objects). A 30-line `awk` extractor handles it. If complexity creeps, fall back to a `jq` invocation gated by `command -v jq` (graceful degradation: warn-and-skip if `jq` is absent and the JSON is too complex for awk — this should never happen with the v0.9.2 catalog shape). |
| **Hook-blocking semantics ambiguity (F-012).** A PreToolUse hook on `git push` may not visibly block in all Claude versions. | Medium | Low | Ship advisory-only in v0.9.2 (per AC-023 wording). Mark as Should, not Must. Upgrade to hard block in v0.10 once F-012 lands runtime-visible feedback. |
| **Catalog drift between markdown and JSON.** Human edits one, forgets the other. | Medium | High | Drift check (`scripts/validate-secret-patterns.py`) runs in CI. PR fails if the two disagree. |
| **Trailer override accidentally bypasses real findings.** Developer sets `[ADD-SECRET-OVERRIDE: SEC-001]` once, then forgets to remove it on a later commit that contains a real AWS key. | Medium | High | Override is per-commit (the trailer lives in the commit message; new commits start clean). Document this clearly. The interactive `--allow-secret` confirmation phrase remains the deploy-time human gate (parent spec AC-016). |
| **Codex runtime path differences.** `${ADD_RUNTIME_ROOT}` is the neutral variable from the plugin-family spec; until that lands, `${CLAUDE_PLUGIN_ROOT}` is the substitute on Claude. | Low | Low | Use `${CLAUDE_PLUGIN_ROOT}` in the deploy/verify edits for now (matches the existing pattern in `core/skills/deploy/SKILL.md` line 102). When the neutral path-var work lands in a later cycle, those references update mechanically via compile.py substitution. |
| **`.secretsignore` syntax mismatch.** Users expect `.gitignore` semantics; `git check-ignore` is the easiest correct implementation but requires a git repo. | Low | Low | Prefer `git check-ignore -v -n -- <path>` when in a repo; fall back to a literal-glob matcher otherwise. Document the fallback's limits (no `**`, no negation). |

## 9. Rollback Strategy

All changes are additive except the three SKILL/rule edits, which insert new sections or rewrite a single bullet. If the executable scanner proves too noisy in user feedback after ship:

1. Revert the deploy SKILL.md change to restore the prose-driven gate (the `core/skills/deploy/SKILL.md` Step 1.5 from before this PR).
2. Revert the verify SKILL.md change to remove Gate 4.6.
3. Revert `core/rules/secrets-handling.md` to point back at the deploy SKILL.md prose.
4. Leave `core/lib/scan-secrets.sh`, `core/security/secret-patterns.json`, the drift validator, and the test suite in place — they are no-harm-done if unused, and re-enabling enforcement is a one-line skill edit.
5. Hooks.json change is removed in the same revert.

The fixture suite, catalog JSON, and validator stay because they document and protect the catalog regardless of whether enforcement is wired into a skill.
