## Executive Summary

ADD at v0.9.4 is a more credible product than its public surface implies. The methodology core is broad and internally coherent (27 skills, 19 rules, two runtimes, GPG-signed releases, 12-job CI matrix, ~120 fixture tests). M3 pre-GA hardening shipped cleanly. But:

1. **Only ~1.5 of 6 of ADD's own published GA criteria are met** (per `.add/config.json:next_promotion_criteria`).
2. **Six release-note claims are partially inert in real use** (telemetry contract without consumer emission, Codex hook stderr never reaches the agent, AGENTS.md regen ritual missed since v0.9.1, marketplace install has zero CI smoke).
3. **The market has shifted faster than ADD has shipped.** Cloud agents (Claude Code Routines, Cursor cloud), visual multi-agent UX, benchmarked memory layers, skill task-evals + security scoring, and a spec-driven branding war are all 2026-era developments where ADD is invisible or behind.
4. **ADD's moat is real but narrow.** The maturity ladder (poc → alpha → beta → ga) is genuinely unique and directly maps to what the Anthropic 2026 Agentic Coding Trends Report names THE problem of 2026: the trust gradient. Most other ADD differentiators (hooks, skill orchestration, AGENTS.md generation) have been absorbed by host runtimes or competing tools.

**The path to v1.0 in ≤6 versions is not more features.** It is truth-pass + closing the GA criteria gates + reframing the methodology around the moats the market validates + explicit "stop competing, start integrating" decisions for capabilities the host runtimes now do better.

A credible v1.0 ships in **6 releases** (v0.9.5 → v1.0.0) over **8–10 weeks**, with a calendar gate at ~2026-06-22 (60-day beta-stability requirement). The earliest plausible v1.0 tag date is **2026-06-25** assuming no slip.

Stop fighting on absorbed surfaces.**

But: 60 days is the minimum interval for users to actually file regressions, and tagging GA before any beta-cycle bug-find is precisely how projects ship "GA" releases that are really still beta.

**Recommendation: hold the line on 60 days.**

## Part 4 — Capability Family Architecture (v1.0 and Beyond)

The user's stated direction: ADD evolves into "a family of loosely coupled capabilities that enhance agent driven development." Synthesis of Swarm 3 § F + Swarm 2 § C + the M3 milestone "Out of Scope" deferrals.

### What stays IN ADD (the canonical core)

These align with ADD's moats, are unique in the market, and are durable across host-runtime evolution:

1. **Methodology lifecycle artifacts** — spec, plan, cycle, retro, milestone. These ARE the methodology.
2. **Maturity ladder** (poc → alpha → beta → ga) — unique to ADD; trust-gradient implementation.
3. **Maturity-aware rule cascade** — load only the rules appropriate to the current maturity level.
4. **Sub-agent role definitions** (test-writer / implementer / reviewer / verify) — methodological scaffolding even when host runtimes ship sub-agent dispatch primitives.
5. **TDD-deletion guardrail / test-count gate** — load-bearing trust mechanism, demonstrably unique.
6. **Swarm worktree pattern** + a NEW live status surface (the gap Swarm 3 § A2 named).
7. **AGENTS.md generation** with spec-lifecycle hooks (Swarm 3 § A8 — make it living, not one-shot).
8. **Cross-project learnings library schema** — canonical schema, regardless of storage backend (engine becomes pluggable per § F2).

### What goes OUT (delegate / integrate)

Capabilities the market does better as standalone tools or where host runtimes have caught up:

1. **Persistent memory engine** — Mem0 / MemPalace / Letta / Zep. ADD provides an **adapter**, not an engine. Default fallback: keep `library.json` as a deterministic versionable cache.
2. **Eval harness infrastructure** — LangChain Deep Agents Eval, Tessl Task Evals, Braintrust. ADD plugs in with a thin `core/evals/` directory of behavioral signatures; runs through whichever harness the user already has.
3. **Telemetry backend** — OTel emission stays in ADD; storage/UI is Honeycomb/Langfuse/Helicone. Don't build a backend.
4. **Skill registry** — Tessl. ADD publishes there; doesn't host.
5. **MCP tool surface** — git, filesystem, web fetch. ADD doesn't ship MCP servers for primitives that already exist; uses what's there.
6. **Computer use / visual feedback** — Anthropic's beta + Cursor's design mode. ADD invokes when at beta+ maturity; doesn't reimplement.
7. **Code search / repo-map** — Aider/Cody/Augment/Sourcegraph. Host responsibility.
8. **Cloud execution / scheduling** — Claude Code Routines / Cursor cloud / GitHub Actions. ADD emits **descriptors** (Routine YAML, GitHub Action workflow, cron expression) so users on any runtime can pick up `/add:routine`-driven scheduled cycles.

## Part 5 — What We Stop Doing (Swarm 3 § D)

Capabilities to actively retire because the market has moved past them or the host runtimes now do them better:

1. **Bespoke hook system where it duplicates host hooks.** Both Claude Code (PostToolUse with `duration_ms`, etc.) and Codex CLI (stable hooks observing MCP tools and apply_patch) ship robust native hooks. Audit `runtimes/{claude,codex}/hooks/` for what's still load-bearing vs. redundant.
2. **Hand-rolled cross-project learning storage.** `~/.claude/add/library.json` was useful in 2025. Now Mem0 / MemPalace / Letta exist as benchmarked engines. Make storage pluggable; default to `library.json` cache + memory-engine-of-choice MCP adapter.
3. **Manual SVG infographic regeneration.** Auto-generate from a data file; stop hand-tuning offsets. Closes part of F-019.
4. **Rules as the only behavioral surface.** Position rules as one layer among (Skills, Rules, Hooks, AGENTS.md, References). The PR #6 `references:` mechanism is already a step in this direction; lean further.

## Part 6 — Where We Double Down (Swarm 3 § C)

The four moats:

1. **Maturity ladder (poc → ga).** Lead the v1.0 README with this. Make the maturity cascade visible to users via `/add:promote --check` and a rendered cascade diagram in docs. The Anthropic 2026 Trends Report calls trust-gradient THE 2026 problem; the maturity ladder IS the trust-gradient implementation.

This positioning matters because Cursor / Cline / Copilot all have skills; they don't have methodology.

4. **TDD-deletion guardrail.** Document this prominently. Package the test-count gate as a standalone hook that can run alongside any methodology — even users who don't adopt full ADD benefit.

## Part 8 — Open Decisions (Maintainer Input Required)

Strategic forks that need the user's call before v0.10 work can scope:

### D1. Hold or revise `next_promotion_criteria`?

The published GA gates (60-day stability, real install smoke, marketplace approval, 20+ projects) are demanding. They're also defensible. **Recommendation: hold all six.** Lower the bar and the public credibility argument collapses ("ADD relaxed its own GA criteria to ship v1.0").

### D2. F-006/F-007 host-neutral kernel — v1.0 or v1.1?

This is genuinely architectural. Touches every skill body, every rule, `scripts/compile.py`, both adapter YAMLs, plus a runtime-overlay layer that doesn't yet exist. **Recommendation: defer to v1.1.** v1.0 is a credibility release; the host-neutral kernel is the right next milestone (M4) but trying to land it in v1.0 displaces the install-smoke + Codex-coherence work that v1.0 actually needs. F-006/F-007 in v1.1 lets v1.0 ship cleanly and v1.1 ship as the architectural milestone with proof (Cursor/Cline adapters).

### D3. F-012 hook-feedback channel — ADD-side adapter or document the limitation?

ADD-side polling adapter is doable. Whether it's clean depends on Codex CLI session lifecycle (research not yet done). **Recommendation: spike option (a) in v0.10; if research shows brittleness, fall back to (b) honest docs and accept warn-only on Codex through v1.0.** Either way, AC-027 makes the difference visible to users.

### D4. ADD as MCP server — v1.0 or v1.1?

Exposing ADD as MCP server is mitigation E5 (MCP-server explosion). It's also a v1.1 capability that doesn't gate GA. **Recommendation: v1.1.** Don't add scope to v1.0.

### D5. Brand strategy — ADD-the-plugin or ADD-the-methodology?

Spec Kit + DeepLearning.AI + JetBrains + Anthropic course materials = an industry consolidation around named methodology brands. ADD is currently invisible in this discourse. Two paths:

- **(a) Single brand: ADD-the-plugin.** Marketing focuses on the plugin install; methodology is implementation detail.
