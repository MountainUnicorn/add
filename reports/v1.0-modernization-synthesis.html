ADD → v1.0 GA: Modernization Synthesis

Riding Claude Workflows & modern Codex — closing GA blockers — with a valuation per change

Date: 2026-06-14 Current: v0.9.5 (beta) Target: v1.0.0 GA (~2026-07-11) Sources: 4 research swarms + repo audit

The one-paragraph read

Two ecosystem shifts now bear directly on ADD's identity. Claude Code shipped Dynamic Workflows — a deterministic JS runtime that fans out subagents (parallel()/pipeline(), schema-validated output, budgets, worktree isolation) — which structurally replicates ADD's hand-rolled swarm orchestration. Codex deprecated custom prompts in favor of Skills + a .codex-plugin format that mirrors Claude's. Net: ADD should stop owning orchestration mechanics and double down on its durable moat — maturity-aware methodology that rides on top of both runtimes' native primitives. Meanwhile, three small-but-real GA blockers (a red CI on main, open release-tooling bug #18, deprecated CI action) must clear before the v1.0 tag is credible.

How "valuation" is scored

Each recommendation carries a Value Score / 100 = GA-impact weighted against effort and strategic durability. Inputs:

  • GA Impact — does it gate, harden, or merely polish the v1.0 tag?
  • Effort — S (<½ day) · M (1–3 days) · L (multi-day / cross-runtime).
  • Durability — does it survive host runtimes absorbing features (the central market risk, E1/E8)?

Priority chips: P0 blocks the tag · P1 ship before GA · P2 GA-window nice-to-have · P3 post-GA.

Track A — Workflows vs. Swarms (the headline)

ADD's swarm is a formal feature: core/references/swarm-protocol.md + the autoloaded core/rules/agent-coordination.md, with maturity-scaled WIP limits (poc=1 → ga=5), worktree/file-reservation isolation, and .add/swarm-state.md coordination. Claude's native Workflows now provide the orchestration mechanism for free. The move is to delegate mechanism, keep methodology.

Overlap verdict: Native Workflows make ADD's manual fan-out + in-conversation synthesis redundant. What stays uniquely ADD: maturity-aware concurrency policy, role briefs (test-writer / implementer / reviewer / verify), trust-but-verify merge gates, and the spec→TDD→retro lifecycle the fan-out serves. Rebuild swarm-protocol as a thin policy layer that emits / configures Workflows, not a parallel orchestrator.
A1 · Workflows alignment

Reframe the swarm protocol as a layer over native Workflows

Rewrite swarm-protocol.md + agent-coordination.md to (a) detect Workflows availability, (b) express maturity WIP limits as Workflow concurrency/budget config, and (c) reserve ADD's prose for role briefs and merge/verify gates — not for re-explaining how to spawn parallel agents. Removes maintenance of a mechanism the host now owns and resolves the E1 "native absorption" risk head-on.

What this does not recommend

  • Don't rebuild orchestration from scratch. Workflows exist; wrap them (A1/A2), don't compete.
  • Don't add net-new features to reach v1.0. The roadmap is explicit: GA = truth-pass + closing the six published criteria, not feature count. Most Track D items are v0.10/v1.1.
  • Don't defer the host-neutral kernel (F-006/F-007) into v1.0. Decision D2 stands — that's architectural v1.1 work; v1.0 is a credibility release.
  • Don't rename to dodge the "ADD" collision. Brand split is a post-v1.0 marketing track (D5); ship the tag clean first.
